## Vision

### 10x Check

The current plan tests one question: "Will members pay?" The 10x version doesn't just
answer the hypothesis — it generates momentum that makes the second gym easier than the first.

The 10x sprint looks like:
- 1 gym goes live. 20 members join (not 5). 10 pay.
- The gym owner's Instagram Reels show 30-second highlights from MATPEAK, with a watermark
  link. 3 new members sign up from Instagram without being personally invited.
- One of those new members DMs: "Is there a Muay Thai gym I can watch on here?"
  MATPEAK is now a discovery product before you built the discovery layer.
- A second gym owner texts the first: "I heard you did this. Can I try?"

The difference between 5 members and 20 is not 4x more features. It's two mechanics:
(1) auto-clips that make the gym owner's Instagram into free MATPEAK advertising, and
(2) replay chapters that make MATPEAK replays worth ₹300/month vs watching free on YouTube.

### Platonic Ideal

The best version of MATPEAK is a platform gym owners send members to BEFORE Instagram,
not instead of it. MATPEAK watermarks every clip, so every gym owner's Instagram becomes
free MATPEAK advertising. Members follow the link, subscribe, and the gym owner earns.
Instagram gets reach; MATPEAK gets conversions. The gym never has to choose between
"free marketing" and "paid access" — both happen simultaneously.

12-month north star: 50 gyms, 2,000 subscribers, ₹600/month average plan = ₹1.2Cr ARR.
The top 5 gyms on the platform are earning more from MATPEAK remote students than from
their physical walk-in fees.

## Accepted Scope (added to this plan)

1. **Auto-clip feature (v0):** After every stream ends, MATPEAK generates a 60-second
   highlight clip (first minute of the class). No watermark in v0 — the gym owner receives
   a download link in their dashboard and shares manually to Instagram. Clips are served via
   Mux's clip API (Mux-native URL, no re-upload to Supabase storage). MATPEAK watermark
   added in v0.1 via ffmpeg overlay after the v0 pilot validates the feature is used.

   Note: the viral advertising mechanic ("MATPEAK watermark on every clip") is a v0.1
   benefit. v0 delivers the clip and tests whether gym owners share it at all.

2. **Replay timestamp chapters:** Gym owner can add, edit, and delete timestamped notes on
   any replay from the gym dashboard ("Triangle choke at 23:41", "Guard passing drill at
   45:00"). Members see clickable chapter markers in the Mux player during replay. Chapter
   data stored in a `replay_chapters` table linked to the stream record.

   RLS policies: INSERT/UPDATE/DELETE restricted to authenticated gym owner where stream
   belongs to their gym. SELECT open to authenticated members with an active membership.
   `created_by` column set server-side to `auth.uid()` — not user-settable.

   Cue-points format for Mux Player: `[{ time: 1421, value: "Triangle choke" }]` where
   `time` is seconds from stream start. Pass as JSON-stringified attribute on `<mux-player>`.

   API endpoints: POST /api/chapters (create), PUT /api/chapters/[id] (edit),
   DELETE /api/chapters/[id] (delete). Max 50 chapters per stream enforced at API level.

3. **Gym owner waitlist CTA:** A minimal form on the landing page: "Own a gym? Apply to
   become a MATPEAK partner." Collects name, gym name, discipline, city, contact (phone
   or email). Stores to a `gym_waitlist` table. Goes live on matpeak.com after the first
   successful pilot class (deploy as a feature flag or just push post-pilot).

   Duplicate guard: unique constraint on `contact` column; API returns 409 "already
   registered" on duplicate, frontend shows "You're on the list!" rather than an error.

   Founder notification: on INSERT, call Resend API (via `RESEND_API_KEY` env var) to
   send an email to `ADMIN_EMAIL`. Use a simple Supabase edge function or a Next.js
   API route at `/api/gym/waitlist` that does the insert + email in one call.

   Spam protection (rate limiting + honeypot): add a Supabase DB rate limit of 10 submissions
   per IP per hour — before INSERT, query `COUNT(*) FROM gym_waitlist WHERE submitted_ip = req_ip
   AND created_at > now() - interval '1 hour'`; if ≥ 10, return 429. `submitted_ip INET` added
   to gym_waitlist schema. No new vendor or env vars needed (uses existing Supabase connection).
   Also add a hidden `website` field to the form (invisible to humans via
   `tabIndex={-1} style={{display:'none'}}`); if non-empty in POST body, return 200 silently
   without inserting to DB or sending email.

## Platform Migration Prerequisite (Cloudflare Stream)

**D2.2 (auto-clip) and D2.3 (replay chapters) cannot be built until the codebase completes
its migration from Mux to Cloudflare Stream.** D2.5 (waitlist) is unrelated to video infra
and can be built in parallel immediately.

**Ingest:** WHIP (WebRTC HTTP Ingestion Protocol) via Cloudflare live inputs.
**Playback:** HLS (`customer-*.cloudflarestream.com/{uid}/manifest/video.m3u8`) or WHEP.
**Recording:** Cloudflare auto-creates a VOD recording from every live stream — no manual
  export needed.

**Codebase changes required:**
1. Remove `@mux/mux-node` dependency; call Cloudflare Stream REST API via `fetch` (no dedicated SDK needed).
2. Replace `/api/gym/create-stream`: create Cloudflare live inputs (`POST /accounts/{CF_ACCOUNT_ID}/stream/live_inputs`).
   **Critical:** live input MUST be created with `recording.mode = "automatic"` — without this,
   Cloudflare does not store recordings and `cf_video_uid` is never populated. The entire sprint
   depends on recordings being available.
3. Replace `/api/webhooks/mux/` → `/api/webhooks/cloudflare/` with Cloudflare webhook signature verification (`CF_STREAM_WEBHOOK_SECRET`).
4. Replace Mux Player (`@mux/mux-player-react`) with Cloudflare Stream player (`@cloudflare/stream-react`).
5. Update `StreamSetupPageClient.tsx`: show the Cloudflare RTMP or WHIP ingest endpoint + stream key.

**Schema migration (`cloudflare_migration`):**
- `gyms`: add `cf_live_input_uid TEXT` (replaces `mux_live_stream_id`). Keep `stream_key` (Cloudflare also uses one for RTMP).
- `sessions`: add `cf_video_uid TEXT` (Cloudflare recording video UID — required for D2.2 clip creation), `duration_seconds INT` (set from Cloudflare recording webhook).
- Old `mux_live_stream_id`, `mux_playback_id` columns: kept for audit, no longer written post-migration.

**New environment variables (add to Vercel + local `.env`):**
- `CF_ACCOUNT_ID` — Cloudflare account ID
- `CF_API_TOKEN` — Cloudflare API token with Stream write permissions
- `CF_STREAM_WEBHOOK_SECRET` — Cloudflare webhook signing secret (Cloudflare dashboard → Stream → Webhooks)
- `NEXT_PUBLIC_CF_CUSTOMER_SUBDOMAIN` — for constructing `customer-{subdomain}.cloudflarestream.com` playback URLs

**Webhook correlation (recording ready):**
When a live stream ends, Cloudflare fires an encoding-complete webhook for the recording.
Look up gym by `cf_live_input_uid` → find most recently ended session → store
`cf_video_uid = data.uid`, `duration_seconds = data.duration`, set `replay_url` to HLS URL.

**Acceptance test (migration gate — run before D2.2/D2.3 sprint work):**
- Start a WHIP or RTMP stream to the Cloudflare live input. Confirm member watch page shows live stream.
- End the stream. Confirm `sessions.cf_video_uid` is populated within 5 minutes and replay plays.

## Implementation Notes for Accepted Scope

### Auto-clip (D2.2)

**Prerequisite:** Cloudflare migration complete; `sessions.cf_video_uid` populated from
the recording webhook before this feature is functional.

**Async two-step flow (Cloudflare-specific):**
1. Cloudflare fires a recording-ready webhook when the live stream recording finishes encoding.
   **IMPORTANT: verify exact Cloudflare event type string against Cloudflare Stream webhook docs
   before implementing** — do NOT trigger on `stream.live_input.disconnect` (recording still
   processing). Use the event that fires when the recording video is ready for playback.
   Handler: look up gym by `cf_live_input_uid` → find most recently ended session → store
   `cf_video_uid = data.uid`, `duration_seconds = data.duration`, `replay_url = HLS URL`.
   Then call the Cloudflare clip API (step 2).
2. Handler calls `POST /accounts/{CF_ACCOUNT_ID}/stream/clip` (SYNCHRONOUS — returns new UID):
   ```json
   {
     "clippedFromVideoUID": "<session.cf_video_uid>",
     "startTimeSeconds": 0,
     "endTimeSeconds": 60
   }
   ```
   Immediately store `sessions.clip_video_uid = response.uid` and `clip_status = 'pending'`.
   If POST returns non-2xx: set `clip_status = 'failed'` immediately; log to Vercel.
3. Cloudflare encodes the clip (async).
4. Cloudflare fires a second encoding-complete webhook for the clip.
   Handler looks up session by `sessions.clip_video_uid = data.uid` (direct UID match — no
   meta parsing needed). Sets `clip_url` (HLS URL) and `clip_status = 'ready'`.

**Correlation approach:** The Cloudflare clip API returns the new clip UID synchronously.
Store it immediately in step 2. Use it to match the step-4 webhook. This avoids relying on
`data.meta` being preserved in the webhook payload (Cloudflare does not explicitly document
this guarantee).

**Idempotency:** At the start of the recording webhook handler, check if `session.cf_video_uid`
is already set. If yes, return 200 immediately (already processed — Cloudflare retry). Only
proceed to clip API call if `cf_video_uid` is NULL. Prevents duplicate clip creation on webhook
retry.

**Webhook endpoint:** `/api/webhooks/cloudflare/` (replaces `/api/webhooks/mux/`).
Verify webhook signature: Cloudflare sends HMAC-SHA256 of the raw request body in the
`Webhook-Signature` header, signed with `CF_STREAM_WEBHOOK_SECRET`. Compute
`HMAC-SHA256(rawBody, CF_STREAM_WEBHOOK_SECRET)` and compare with timing-safe comparison
(`crypto.timingSafeEqual`). Return 400 on mismatch — never 200-silently. If verification
fails silently (wrong header name or algorithm), ALL webhooks are rejected and the entire
auto-clip pipeline breaks without error.

**Short stream guard:** `data.duration < 70` → skip clipping, return early. (70s buffer:
clip API needs at least 60s of content with headroom for encoding edge cases.)

**Error/retry:** If clip encoding webhook never arrives after >30 minutes, `clip_status`
stays `'pending'`. For v0: gym dashboard note "Clip still processing? If nothing appears
after 30 minutes, click Retry." Retry button calls `/api/gym/retry-clip` → re-calls clip
API → stores new `clip_video_uid` → resets `clip_status = 'pending'`.
No separate `clip_errors` table — log to Vercel function logs only.

**Historical sessions:** Sessions created before the Cloudflare migration (still have
`mux_playback_id`, no `cf_video_uid`) cannot be auto-clipped. The "Retry clip" button
provides manual fallback if `cf_video_uid` is populated later.

**Schema:** Add to `sessions` table (migration: `add_clip_fields_to_sessions`, runs AFTER `cloudflare_migration`):
- `clip_url TEXT`
- `clip_status TEXT DEFAULT 'pending'` (values: pending | ready | failed)
- `clip_banner_dismissed BOOL DEFAULT FALSE`
- `clip_video_uid TEXT` (Cloudflare clip video UID — stored for future reference)

Note: `cf_video_uid` and `duration_seconds` are added by `cloudflare_migration`, not here.

**Naming:** Call this a "preview clip" (not "highlight clip") in all UI copy — the first 60
seconds of a combat sports class is warmup/admin, not highlights. "Preview clip" accurately
sets the gym owner's expectations.

**Stripe note:** Stripe is integrated in the codebase but NOT used for v0. Member payments
are collected out-of-band by the gym owner via their own Razorpay link (shared via WhatsApp).
The clip pipeline does not touch Stripe code.

**v0 delivery:** Download link only. No watermark. Gym owner downloads and shares to
Instagram manually. Watermark deferred to v0.1.

**Dashboard:** "Your clip is ready — Download" banner on gym dashboard home, visible until
dismissed (`clip_banner_dismissed = TRUE`). "Retry clip" button visible when:
- `clip_status = 'failed'`, OR
- `clip_status = 'pending'` AND session ended >30 minutes ago (frontend check:
  `sessionEndedAt < Date.now() - 30 * 60 * 1000`)
This ensures the Retry button surfaces even if the encoding-complete webhook never fires
(wrong event name, Cloudflare error), not just when status is explicitly 'failed'.

**Design spec (anti-slop):** Banner is a `bg-[#1A1A1A] border border-[#FF3B3B]/30 rounded-sm`
card — NOT a toast pill, NOT `rounded-full`. Headline in Bebas Neue: "YOUR CLIP IS READY".
Dismiss is an `×` icon in `text-[#555555]`, top-right corner. "Download" is a `text-[#FF3B3B]`
link, NOT a teal button. "Retry clip" is a ghost button (`border-[#333333] text-[#999999]`).

### Replay Chapters (D2.3)

**New table:** `replay_chapters` (migration: `create_replay_chapters_table`):
- `id UUID PRIMARY KEY DEFAULT uuid_generate_v4()`
- `session_id UUID NOT NULL REFERENCES public.sessions(id) ON DELETE CASCADE`
- `timestamp_seconds INT NOT NULL`
- `label TEXT NOT NULL`
- `created_by UUID REFERENCES public.users(id) ON DELETE SET NULL`
- `created_at TIMESTAMPTZ NOT NULL DEFAULT now()`
- Index: `CREATE INDEX ON replay_chapters(session_id);` — included in migration.

**RLS:**
- INSERT/UPDATE/DELETE → gym owner only (session's gym_id = owner's gym). `created_by` set server-side to `auth.uid()` — not client-settable.
- SELECT → gym owner OR authenticated member with active membership.

**RLS SELECT policy (SQL):**
```sql
CREATE POLICY chapters_select ON replay_chapters FOR SELECT USING (
  EXISTS (
    SELECT 1 FROM memberships m
    JOIN sessions s ON s.gym_id = m.gym_id
    WHERE s.id = replay_chapters.session_id
      AND m.user_id = auth.uid()
      AND (m.current_period_end > now() OR m.free_until > now())
  )
  OR EXISTS (
    SELECT 1 FROM sessions s
    JOIN gyms g ON g.id = s.gym_id
    WHERE s.id = replay_chapters.session_id
      AND g.owner_id = auth.uid()
  )
);
```

**Player approach (Cloudflare Stream):**
Using `@cloudflare/stream-react` (`<Stream src={session.cf_video_uid} controls />`).
For chapter markers, do NOT rely on Cloudflare player-native chapter API in v0 — use a
**custom chapter list** below the player instead:
- Replay page uses `Promise.all([fetchSession(id), fetchChapters(id)])` — parallel fetch,
  not serial, to avoid extra roundtrip latency on Indian mobile networks.
- Render as sorted clickable list: `"23:41 — Triangle choke"` entries (sorted ORDER BY timestamp_seconds ASC at DB level, not JS-side).
- On click: `streamRef.current.currentTime = chapter.timestamp_seconds` via `playerRef` prop.
- Minimal MATPEAK styling: `font-inter text-sm`, red dot marker, `font-bebas` for timestamps.
- This approach is player-agnostic — works regardless of Cloudflare player version.

**Gym dashboard chapter editor:**
- Timestamp input + label input + "Pin" button. Chapters sorted by timestamp with edit/delete.
- Hidden until `sessions.replay_url` is set. Visibility: poll `/api/gym/stream-status?gym_id=X`
  every 30s (already exists) — enable chapter editor when response includes non-null `replay_url`.

**API endpoints:**
- `POST /api/chapters` (body: `session_id`, `timestamp_seconds`, `label`)
- `PUT /api/chapters/[id]` (update label/timestamp)
- `DELETE /api/chapters/[id]`
- `GET /api/chapters?session_id=X` (list for member watch page)
- All write endpoints verify gym ownership via DB: query `sessions.gym_id → gyms.owner_id = auth.uid()`. Never from JWT claims — recent commits proved JWT-based gym ownership is broken in this codebase.

**Edge cases:**
- Chapter on a live session: UI hides editor until `session.status = 'ended'`.
- Max 50 chapters: API returns 422 if exceeded.
- Timestamp > duration: compare against `sessions.duration_seconds`; return 422 if exceeded.
  Skip validation if `duration_seconds` is NULL (recording still processing).
- Duplicate timestamps: allowed.
- Chapter access expiry: chapter SELECT follows the same access model as video access.
  Expired members see the ACCESS LOCKED screen before the chapter fetch is attempted —
  consistent with platform-wide content gating. No special chapter-empty-state handling needed.

**Design spec (anti-slop):** Chapter editor is a `bg-[#1A1A1A] border border-[#333333] rounded-sm`
card. "CHAPTERS" section label uses the red eyebrow-line pattern. Timestamp input is monospace
`font-mono text-sm`, label input is `font-inter`. "Pin" button is `bg-[#FF3B3B] text-white
font-bebas tracking-[1px] rounded-sm`. Chapter list rows have `font-bebas` timestamps + 
`font-inter text-sm` labels. No `rounded-xl`, no teal accents.

### Payment CTA (Pilot Day — NOT a feature, a process)

Payments are out-of-band for v0. But the member needs to know HOW to pay before or after
joining. Implementation: when the gym owner generates a member invite link (Members tab →
Invite), the welcome message template should include the gym owner's Razorpay payment link.
The gym owner pre-fills this once in their dashboard profile (add `razorpay_link TEXT` to
the gym record) and the invite flow appends "Pay to access: [link]" to the message.

This is a 1-hour addition (one new optional field on the gym profile, one line in the invite
flow) that closes the "does a member know where to pay?" gap. Without this, the pilot can't
answer the primary question.

Schema: add `razorpay_link TEXT` to `gyms` table (migration: `add_gym_razorpay_link`).
UI: gym profile edit form → optional "Your Razorpay payment link" field.

These issues were raised by independent review and are NOT resolved. They are documented
here for the engineer's awareness. They do not block the sprint but should be revisited
before v0.1.

**RC1 — Replay chapters scope for demand validation:**
An independent reviewer noted that chapters (D2.3) are a retention feature, not a
conversion feature. The sprint's primary question ("will members pay?") doesn't require
chapters to be answered. Chapters add an estimated 2 days of engineering for a feature
that has no direct signal on conversion rate at pilot scale (1 gym, 5-20 members).
If the sprint window is tight, D2.3 is the first thing to cut. The user chose to include
it; this concern is logged for retrospective.

**RC2 — Cloudflare meta preservation in webhooks:**
It is not explicitly documented that custom `meta` set during clip creation is echoed
back in the encoding-complete webhook payload. The current plan uses UID-based correlation
(not meta) to mitigate this, but the implementer should verify Cloudflare webhook payload
structure with a test clip before pilot day.

**RC3 — Cloudflare recording event name:**
The exact Cloudflare Stream webhook event type string for "recording is ready to clip"
is not confirmed in this plan. DO NOT implement the clip trigger on `stream.live_input.disconnect` —
the recording is still processing at that point. Verify the correct event name in
Cloudflare Stream webhook documentation before writing the handler.

**Architecture decisions locked by this review:**
- Chapter API auth: DB-based gym ownership check only (never JWT claims) — `sessions.gym_id → gyms.owner_id = auth.uid()`
- Cloudflare webhook signature: HMAC-SHA256 of raw body in `Webhook-Signature` header, `crypto.timingSafeEqual` comparison, return 400 on mismatch
- Recording webhook idempotency: check `cf_video_uid` before clip API call; return 200 if already set
- Waitlist rate limit: Supabase DB query (`COUNT submitted_ip last 1h`); no Upstash/Redis needed
- Clip Retry UX: show Retry when `clip_status = 'pending'` AND `session ended > 30min ago` (not only on `failed`)
- Chapter parallel fetch: `Promise.all([fetchSession, fetchChapters])` on replay page
- Chapter access expiry: follows ACCESS LOCKED model (consistent with platform content gating)
- Temporal Interrogation: UID-based clip correlation throughout (meta approach removed); short stream guard = 70s not 60s
